# VPS / VPN / 代理 / 机场 / 节点 / 静态住宅IP / 链式代理学习手册

更新时间：2026-04-02
适用对象：从零到一，想搭建“稳定可维护”的链式代理环境

## 0. 先说边界
- 本文只用于网络与系统工程学习、隐私保护、跨地域业务访问等合规用途。
- 你需要自行遵守所在地区法律法规、服务商条款和目标网站政策。

## 1. 核心概念（先分清名词）

### 1.1 VPS 是什么
- VPS（Virtual Private Server）是云上的一台可远程管理的虚拟服务器。
- 你可以在 VPS 上部署代理服务、VPN 服务、转发服务、监控程序。
- 作用：提供“你可控”的中转点。

### 1.2 VPN 是什么
- VPN（Virtual Private Network）本质是加密隧道，把你和某个网络之间连成一条受保护通道。
- 常见协议：WireGuard、OpenVPN、IPsec。
- 作用：把“本机流量”整体或按规则送入隧道。

### 1.3 代理 是什么
- 代理（Proxy）是“按连接/请求转发流量”的中间层。
- 常见协议：SOCKS5、HTTP(S)、VMess/VLESS、Trojan、Shadowsocks。
- VPN 更像“网络层通道”，代理更像“应用层/会话层转发”。

### 1.4 机场 是什么
- “机场”是中文社区常用叫法，一般指提供订阅链接、多个节点、统一管理面板的代理服务商。
- 你拿到订阅后，客户端可批量导入节点。
- 本质：商业化的代理节点集合。

### 1.5 节点 是什么
- 节点一般指一个可连接的代理入口（某个 IP + 端口 + 协议 + 认证信息）。
- 一个机场通常有很多节点（不同地区、不同线路、不同协议）。

### 1.6 静态住宅 IP 是什么
- 住宅 IP：看起来来自家庭宽带运营商网段，而非数据中心机房。
- 静态住宅 IP：地址相对固定，不频繁变化。
- 常见用途：降低“机房 IP”被风控或限流的概率。
- 风险点：贵、供应质量差异大、合规和来源透明度很关键。

### 1.7 链式代理（Proxy Chaining）是什么
- 链式代理是让流量依次经过多个中转层，例如：
  - 本机客户端 -> A 节点 -> B 节点 -> 目标网站
- 目的通常是：
  - 提高隐私隔离（不同层角色分离）
  - 提高可用性（某层失效时可切换）
  - 让出口特征更符合业务需求（例如稳定地区出口）

## 2. 这些概念之间的关系（记忆版）
- VPS：基础设施（你自己控制的服务器）
- VPN：整条加密通道
- 代理：按连接转发机制
- 节点：一个具体可用代理入口
- 机场：节点供应商
- 静态住宅 IP：一种出口 IP 资源类型
- 链式代理：把多个代理/隧道串起来的架构

## 3. 稳定链式代理的推荐架构

## 3.1 最实用的起步架构（两跳）
- 跳板 1（入口层）：你自建 VPS（可控、可审计）
- 跳板 2（出口层）：高质量商业节点或静态住宅出口
- 本机只连入口层，入口层再转出口层。

优点：
- 你不会把所有信任都给单一供应商。
- 出问题时可快速定位是入口故障还是出口故障。

### 3.2 进阶架构（三层解耦）
- 本机策略层（规则分流）
- 中间转发层（VPS 集群，负责中继与健康检查）
- 出口资源层（多个地区出口：机房 IP + 住宅 IP）

适合：长期稳定使用、需要按业务场景切换出口策略。

## 4. 工具选型建议（实战导向）
- 客户端：sing-box（规则灵活、链路编排能力强）
- 隧道层：WireGuard（性能高、配置相对简洁）
- 代理层：SOCKS5/HTTPS 入站 + 上游代理出站
- 监控：Prometheus + Grafana 或最小化脚本健康检查

思路：
- 用 WireGuard 保障你到入口 VPS 的稳定加密链路。
- 在入口 VPS 或本机 sing-box 中做“上游代理链”和自动故障切换。

## 5. 稳定性的 8 个关键点

### 5.1 DNS 策略
- 使用受信任 DNS（DoH/DoT）并确保 DNS 请求走代理链。
- 避免本地运营商 DNS 直连导致泄漏或污染。

### 5.2 路由与分流
- 国内/本地业务直连，外部业务走代理，避免全量绕路。
- 对关键域名单独指定出口节点（例如办公、代码托管、文档服务）。

### 5.3 健康检查与自动切换
- 每 10-30 秒做延迟和可达性探测。
- 连续失败阈值触发 failover，恢复后再回切（防抖动）。

### 5.4 连接复用和并发控制
- 开启连接复用（mux）但不要过度，避免单链路拥塞放大。
- 合理限制并发，避免把不稳定线路“打爆”。

### 5.5 MTU/MSS 调优
- WireGuard 或隧道链路常见 MTU 问题会导致卡顿和间歇性超时。
- 实践中常从 MTU 1280-1420 之间压测确定最稳值。

### 5.6 时钟同步
- 客户端与服务器保持 NTP 同步，避免 TLS 握手异常。

### 5.7 出口多样化
- 不要只有一个出口供应商。
- 至少准备：主出口 + 同地区备出口 + 跨地区兜底出口。

### 5.8 观测与日志
- 记录：握手失败率、重连次数、平均 RTT、丢包率、出口切换次数。
- 没有观测就无法做稳定性优化。

## 6. 从 0 到 1 的落地步骤

### 第一步：准备资源
- 1 台你自管 VPS（建议先选网络质量稳定地区）
- 1 组高质量出口节点（可先机房 IP，后续再评估住宅 IP）
- 本机安装支持链式出站的客户端（如 sing-box）

### 第二步：先做单跳基线
- 先验证“本机 -> VPS（WireGuard）”稳定。
- 基线指标：连续 24 小时连接不掉线、关键网站访问无明显抖动。

### 第三步：加第二跳
- 在策略层配置“VPS -> 出口节点”。
- 验证是否存在 DNS 泄漏、地理位置异常、TLS 错误。

### 第四步：加自动故障转移
- 配置主备出口和健康检查。
- 设置合理回切策略，避免频繁来回切换。

### 第五步：做分流优化
- 业务域名分组：办公、代码、流媒体、下载。
- 给每组分配最合适出口，减少互相干扰。

### 第六步：持续运维
- 每周复盘日志：延迟曲线、失败原因、切换频率。
- 每月复核供应商质量：丢包、封锁率、可用时长。

## 7. 静态住宅 IP 该不该上
适合场景：
- 业务强依赖“非机房出口特征”，且预算充足。

不建议场景：
- 只是日常学习访问，预算有限。
- 还没建立监控和容灾，先上贵线路意义不大。

建议：
- 先把架构和运维做对，再升级出口资源。
- 住宅 IP 作为“特定业务出口池”，不是所有流量默认出口。

## 8. 常见故障排查清单
- 能连上但网页慢：优先查 MTU、丢包、DNS、出口拥塞。
- 部分网站打不开：查 SNI/TLS、DNS 解析路径、分流规则命中。
- 频繁掉线：查 VPS 网络抖动、客户端保活、NAT 超时、时钟偏差。
- 速度波动大：查是否单节点过载、是否需要多出口负载分担。

## 9. 最小可执行方案（推荐）
- 架构：本机 sing-box -> WireGuard 到自建 VPS -> 上游代理出口池（主备）
- 策略：按域名分流，关键业务固定稳定出口
- 保障：健康检查 + 自动切换 + 日志观测

这个方案的特点是：
- 成本可控
- 可逐步演进
- 稳定性明显优于“单机场单节点直连”

## 10. 学习路线（两周）
- 第 1-3 天：吃透概念，完成单跳基线。
- 第 4-7 天：完成两跳链路与故障切换。
- 第 8-10 天：完成分流策略和 DNS 防泄漏。
- 第 11-14 天：建立日志看板并进行一次稳定性复盘。

## 11. 常见认知澄清（高频误区）

### 11.1 VPS 和“我自己的服务器”有什么区别
- VPS 是云厂商提供的虚拟机，通常自带机房网络、公网 IP、按月计费。
- 你自己的服务器可能在家里、办公室或机柜中，公网可达性、带宽和稳定性取决于你本地网络条件。
- 二者都可以跑代理；区别是部署位置和网络属性，不是“能不能做代理”。

### 11.2 我是不是入口，怎么判断
- 入口的定义是第一跳中转点：你的流量离开本机后，最先进入的那台中转主机就是入口。
- 示例 A：本机 -> VPS -> 出口节点 -> 目标站点。入口是 VPS。
- 示例 B：本机 -> 你家服务器 -> 出口节点 -> 目标站点。入口是你家服务器。
- 示例 C：本机 -> 商业 VPN -> 你家服务器 -> 出口节点 -> 目标站点。入口是商业 VPN 的第一跳。

### 11.3 VPS 不就是代理吗
- 不是。VPS 是基础设施（服务器），代理是运行在其上的服务（软件）。
- 只有在 VPS 上部署并启用代理程序后，它才会充当代理节点。
- 同一台 VPS 也可以同时运行网站、数据库、监控等其它服务。
