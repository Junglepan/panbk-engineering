# 《Agent》阅读总结（tw93，2026-03-21）

原文链接：https://tw93.fun/2026-03-21/agent.html

## 一、核心判断

文章强调：Agent 的“智能感”并不主要来自模型本身，而来自完整工程系统的设计。一个可复用的 Agent，需要把模型、工具、上下文、验证和回退机制组织成稳定闭环。

## 二、关键观点

1. Agent 本体很小，系统工程很大
- 语言模型 + 工具调用循环只是内核。
- 真实效果取决于外层 harness：任务分解、状态跟踪、失败恢复、质量评估。

2. 上下文管理决定成败
- 不能把所有规则和技能一次性塞进上下文。
- 应采用“常驻最小规则 + 按需加载技能 + 动态注入任务信息”的分层策略。

3. 工具越多不一定越好
- 工具定义和工具输出都会污染上下文。
- 需要压缩工具返回、控制工具数量、把结果结构化，减少噪声。

4. 验证机制是 Agent 的护栏
- 仅靠“模型说完成了”不可靠。
- 必须外接可执行验证（测试、lint、脚本、检查项），用证据驱动完成判定。

5. 记忆需要独立设计
- 会话内上下文不是长期记忆。
- 长期偏好、项目决策、关键约束应外置到 memory 层，并可按需注入。

## 三、对工程实践的启发

1. 先搭框架再追模型
- 在现有模型能力下，先把流程闭环做扎实：计划、执行、验证、回滚。

2. 把“规则文档”变成“可执行约束”
- 能落到脚本和 CI 的规则，不只写在说明里。
- 让 Agent 在真实工程环境中被自动约束。

3. 设计可观测性
- 对每一步保留最小必要日志：输入、动作、结果、失败原因。
- 便于跨会话接力和问题追踪。

## 四、可落地模板（可直接用于项目）

1. Soul（项目常驻）
- 文件：`AGENTS.md`
- 内容：流程顺序、硬约束、验收标准、禁止项。

2. Memory（长期记忆）
- 文件：`MEMORY.md` 或 `docs/memory.md`
- 内容：长期偏好、架构决策、历史坑点、常用命令。

3. Skills（按需能力）
- 路径：`.agents/skills/...`
- 策略：技能目录可见，正文按需加载。

4. Verifier（完成判定）
- 路径：`scripts/verify.sh` + CI
- 策略：完成前强制通过可执行检查。

## 五、一句话总结

这篇文章的核心是：把 Agent 当作“软件系统”而不是“会聊天的模型”，重点投入在上下文分层、工具治理、验证闭环和可观测性上，收益会比单纯更换模型更稳定。
