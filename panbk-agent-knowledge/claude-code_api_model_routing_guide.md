# Claude Code、API 与模型调用差异详解

## 1. 文档目标

这份文档总结了你这次遇到的问题本质：

- 为什么“给 Claude 配其他厂商 API”常常很简单
- 为什么“在 Claude Code 里用 Codex/OpenAI 模型”通常更复杂
- Proxy/Router 工具到底做了什么
- 如何用一套稳定的心智模型定位问题

---

## 2. 先分清三个层：模型、API、客户端

很多使用问题都来自把这三件事混为一谈。

### 2.1 模型（Model）

真正生成内容的引擎。例如 Claude 系列、GPT/Codex 系列等。

### 2.2 API（Protocol/API Surface）

调用模型的“协议与接口形状”：

- 请求地址（Base URL）
- 鉴权方式（API Key/Header）
- 请求字段（messages、temperature、max tokens 等）
- 流式输出格式（SSE chunk/event 结构）
- 工具调用字段（tool/function call schema）

### 2.3 客户端（Client/Agent）

真正发请求的软件。例如 Claude Code、IDE 插件、SDK 应用。

客户端通常“默认只会说一种方言协议”。当目标模型不在同一生态，就需要翻译层。

---

## 3. 为什么“其他厂商 API 看起来更简单”

核心原因通常不是能力更强，而是做了兼容策略：

1. 提供 OpenAI-compatible 接口
2. 只需改三项：`base_url`、`api_key`、`model`
3. 你的使用场景可能只涉及基础文本补全，未触发复杂能力

因此体验是“改个地址就能跑”。

---

## 4. 为什么 Claude Code 接 Codex 更难

Claude Code 默认围绕 Anthropic 生态设计。你希望它使用 OpenAI/Codex 时，会出现协议和语义错位。

常见差异点：

1. 请求字段映射差异
2. 流式事件格式差异
3. 工具调用结构差异
4. 错误码与重试语义差异
5. 模型命名和能力边界差异

所以不是单纯“填错 Key”，而是需要一层协议转换。

---

## 5. Proxy 与 Router 的作用原理

## 5.1 Proxy（代理）

作用：做“协议翻译”。

调用链：

`Claude Code -> Proxy(Anthropic风格入口) -> OpenAI/Codex`

工作内容：

- 转换请求体字段
- 重组流式响应事件
- 映射错误结构
- 处理模型名映射

优点：

- 上手快
- 配置少

缺点：

- 高级场景兼容边缘较多（工具调用、复杂长会话）

## 5.2 Router（路由器）

作用：在 Proxy 基础上增加“策略调度”。

常见策略：

- 按任务规模路由不同模型
- 主模型失败自动 fallback
- 成本优先或质量优先策略

优点：

- 灵活
- 可做成本/稳定性优化

缺点：

- 配置和排障更复杂

---

## 6. 常见配置项背后的真实含义

### 6.1 `ANTHROPIC_BASE_URL`

让 Claude Code 把请求发到本地代理，而非默认官方端点。

### 6.2 `OPENAI_API_KEY`

代理转发到 OpenAI/Codex 时使用的上游鉴权。

### 6.3 `OPENAI_BASE_URL`

上游服务地址（官方或兼容服务商）。

### 6.4 模型映射（Model Mapping）

把客户端请求中的模型位（或档位）映射为上游真实模型名。

## 7. 一张图看懂完整调用链

```text
[Claude Code]
   |
   | Anthropic-style request
   v
[Local Proxy/Router]
   | 1) auth verify
   | 2) schema transform
   | 3) model routing/mapping
   v
[Upstream Provider API]
   |
   | real inference
   v
[Target Model Runtime]
   |
   | stream/result
   v
[Local Proxy/Router] -> reformat -> [Claude Code]
```

---

## 8. 最容易踩坑的点（高频）

1. “兼容”不等于“行为完全一致”
2. 流式输出能连上但事件结构不匹配
3. 模型名可用但能力不匹配（上下文长度、工具能力）
4. 看似模型问题，实际是代理层重试/超时策略问题
5. 代理监听范围过大（`0.0.0.0`）带来安全风险

---

## 9. 分层排障方法（实战）

按层定位，不要混查：

### 9.1 Client 层

检查：环境变量是否生效、是否真的请求到代理地址。

### 9.2 Adapter 层（Proxy/Router）

检查：

- 启动日志
- 请求是否进入
- 转换后 payload 是否符合上游要求
- 流式转发是否完整

### 9.3 Provider API 层

检查：

- Key 是否有效
- 模型名是否可用
- 配额/限流

### 9.4 Model Runtime 层

检查：

- 模型是否支持所需能力
- 输出是否存在系统性截断/异常

排障原则：

- 先确认“通路是否通”
- 再确认“协议是否对”
- 最后才调“提示词和策略”

---

## 10. 选型建议（结合你的目标）

你的目标是：`在 Claude Code 里使用 Codex`。

建议路线：

1. 先选轻量 Proxy 跑通基础链路
2. 确认流式与稳定性后，再升级到 Router 做策略优化
3. 先求“稳定可用”，再追求“最省成本/最优质量”

判断一个仓库是否值得用：

- 最近提交是否持续
- Issue 响应速度
- 是否明确支持 Claude Code 场景
- 是否有可复现的最小配置示例

---

## 11. 最终心智模型（记住这四层）

任何模型接入问题都能放进这四层：

1. Client（谁发请求）
2. Adapter（谁翻译/路由）
3. Provider API（谁提供协议入口）
4. Model Runtime（谁真正推理）

你以后只要按这四层看日志，定位效率会高很多。

---

## 12. 一句话总结

你这次“用不了”的根因不是 API Key 本身，而是客户端生态与目标模型生态不一致，需要代理/路由器做协议翻译和能力对齐。

---

## 13. Router 调 Codex API 和“订阅反代”是不是一回事

不是一回事。`Router` 是技术组件，“订阅反代”是鉴权来源与计费通道的用法。

### 13.1 `Router + 正规 API Key`（推荐）

- 使用正式 API key 调用上游 API
- Router 负责路由、转换、降级、重试
- 本质仍是标准 API 调用链路
- 稳定性和可维护性更好

### 13.2 “订阅反代”

- 常见做法是复用网页/客户端登录态凭证，把订阅通道当 API 通道
- 也可能经过 Router/Proxy，但身份并非标准 API key
- 容易触发会话失效、风控限制、调用不稳定
- 可能违反平台服务条款

结论：你可以“用 Router 做正规 API 转发”，这不是订阅反代；也可以“用 Router 包装订阅反代”，这是高风险路线。

---

## 14. 你之前失败的高概率根因

如果你当时用的是订阅身份而非 API key，失败通常由以下原因导致：

1. 鉴权类型不匹配：上游预期 API key，却收到会话凭证
2. 会话时效问题：订阅会话 token 过期或被刷新
3. 协议语义不一致：客户端行为像 API 程序调用，不像真人交互
4. 限流/风控差异：订阅链路并不按 API 并发和吞吐设计

---

## 15. 为什么平台通常能检测到“订阅反代”使用

并不是“猜测你在做什么”，而是链路特征天然可识别：

1. 鉴权特征：token 类型、header 组合和签名习惯与 API key 调用不同
2. 请求指纹：User-Agent、请求节奏、并发模式、会话生命周期异常
3. 协议行为：流式事件消费、重试模式、错误处理与官方客户端行为偏离
4. 网络特征：IP/地区/设备指纹与正常订阅使用画像不一致

这些信号叠加后，系统可以做风险识别、限流、挑战验证或直接拒绝。
