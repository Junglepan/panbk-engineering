migrate-to-codex 全面分析

日期：2026-05-12
对象：/Users/panbk/.codex/skills/migrate-to-codex

migrate-to-codex 是一个把 Claude Code 生态中的配置、指令、skills、slash commands、subagents、MCP 与部分 hooks 迁移到 Codex 生态的本地迁移器。它的本质不是完全自动等价转换，而是静态扫描、结构化转换、风险标注和验证。

目的与作用

它的主要目标是降低 Claude Code 用户迁移到 Codex 的人工成本，把分散在 .claude/、.mcp.json、.claude.json、CLAUDE.md 等位置的资产转换为 Codex 可识别的文件。

它适合作为迁移第一步：快速盘点资产，批量生成 Codex 基础文件，再由人或 agent 修复语义差异。它不适合作为无人值守的最终迁移器，尤其是权限、hooks、slash command 参数展开、插件、子代理运行模型等部分。

模块职责
SKILL.md定义 agent 使用迁移器时的流程、顺序、报告格式与自愈循环。
references/differences.md记录 Claude Code 与 Codex 的可映射项、部分映射项和不支持项。

plugins.py只报告 Claude plugins / marketplaces 需要人工迁移。

底层实现原理

迁移面抽象

迁移器把 Claude Code 资产拆成 instructions、plugins、hooks、skills and commands、config and MCP、subagents。cli.convert_scope() 以固定顺序调用各 surface 的转换函数，并汇总为统一的 ConversionResult。

数据含义
summary数量统计，例如 skills、subagents、MCP server 数量。
artifacts准备写入 Codex 目标目录的文件、复制项或 symlink。
report_items报告项，说明已转换、需要人工复核或不支持。

Artifact 模型

迁移输出统一表达为 PlannedArtifact。payload 有三种：GeneratedText 写入生成文本，SourceCopy 复制源文件，SourceSymlink 创建 symlink。这让 instructions、skills、config 和 subagents 都能用同一套部署逻辑。

扫描、规划、部署分离

模式作用
--scan-only只列出源目录中存在的支持项。
--plan列出阶段、目标 artifact 和人工复核项，不写文件。
--doctor评估就绪度，统计人工复核、冲突、孤儿文件风险。
--dry-run模拟迁移并输出完整报告。
--validate-target验证已有 Codex 目标目录的格式和基本可用性。

部署策略

默认是 merge：写入计划中的 artifact，保留未被本次计划覆盖的旧 Codex skills / agents。--replace 会删除孤儿生成物，更接近“目标目录等于源目录映射结果”。

可适配内容范围

内容适配程度说明
provider-neutral instructions高可 symlink 到 AGENTS.md。
普通 Claude skills高prompt 与 scripts/、references/、assets/ 可迁移。
MCP stdio/http 基础配置中高command、args、url、env、headers 可迁移。
基础 subagent prompt中高可转为 Codex custom-agent TOML。
model / effort中按模型族做启发式映射。
acceptEdits / readOnly中可映射为 Codex sandbox。

需要人工复核的内容

内容原因
slash commands参数、模板、文件引用、shell 插值运行时不等价。
allowed-toolsCodex skill 没有同等硬权限字段。
subagent tools / disallowedTools只能保留为 prompt guidance。
subagent skillsCodex 没有相同的 spawn-time preload 语义。
hooks生命周期和匹配范围不同。
MCP auth fallback${VAR:-default} default 语义不保留。
Claude-only instructions需要人工改写为 Codex 语义。

不自动迁移的内容

内容原因
Claude pluginsCodex plugin manifest 和布局不同。
plugin marketplaces不自动 fetch、install 或转换 marketplace。
unsupported hook eventsCodex 没有对应生命周期。
HTTP / prompt / agent / async hooks当前转换器只写 command hooks。
subagent background / isolation / maxTurnsCodex custom-agent 文件不表达等价运行模型。
.claude/settings.local.json local-only 语义Codex 项目配置信任模型不同。

各迁移面的策略

Instructions

候选文件包括 .claude/CLAUDE.md、CLAUDE.md、claude.md、AGENTS.md。如果内容没有明显 Claude-only 标记，就创建 AGENTS.md symlink；如果包含 /hooks、.claude/agents/、permissionMode、ExitPlanMode 等标记，就生成副本并追加 MANUAL MIGRATION REQUIRED。

Skills 与 Slash Commands

Claude skills 会转成 .agents/skills/<name>/SKILL.md。目录型 skill 会复制 scripts/、references/、assets/。Claude allowed-tools 会作为提示文字保留，但不是 Codex 权限边界。

slash command 会转成 source-command-* skill，并保留原始 command template。$ARGUMENTS、$1、{{name}}、shell 输出插值和 @path 文件引用都会被标记为人工复核。

MCP 与 Config

.mcp.json / .claude.json 中的 mcpServers 会写入 .codex/config.toml。headers 和 env 会尽量转成 Codex 的 bearer_token_env_var、env_http_headers、http_headers、env_vars 和 env。

只要生成 config，迁移器会加入 personality = "friendly"，以保留接近 Claude Code 的交互风格。

Hooks

只转换 PreToolUse、PostToolUse、SessionStart、UserPromptSubmit、Stop 中的 command hooks。迁移结果写入 .codex/hooks.json，同时在 config 中启用 [features].codex_hooks = true。这是高风险迁移面，因为 Codex hook runtime 和 Claude 不等价。

Subagents

.claude/agents/*.md 会转成 .codex/agents/*.toml。name、description、model、effort、部分 permissionMode 会被结构化映射；skills、tools、disallowedTools 会写入 developer_instructions，但只是 prompt guidance。

Plugins

插件只报告，不自动迁移。原因是 Claude plugins 可能包含 commands、agents、MCP、skills、hooks、marketplaces、output styles 和 provider-specific metadata，不能安全复制为 Codex plugin。

命令与推荐流程

MIGRATE_TO_CODEX='python3 .codex/skills/migrate-to-codex/scripts/migrate-to-codex.py'

$MIGRATE_TO_CODEX --source ~/.claude/ --scan-only
$MIGRATE_TO_CODEX --source ~/.claude/ --target ~/.codex/ --plan
$MIGRATE_TO_CODEX --source ~/.claude/ --target ~/.codex/ --doctor
$MIGRATE_TO_CODEX --source ~/.claude/ --target ~/.codex/ --dry-run
$MIGRATE_TO_CODEX --source ~/.claude/ --target ~/.codex/
$MIGRATE_TO_CODEX --validate-target ~/.codex/

项目级迁移可使用：

$MIGRATE_TO_CODEX --source ./.claude/ --target ./.codex/ --dry-run
$MIGRATE_TO_CODEX --source ./.claude/ --target ./.codex/
$MIGRATE_TO_CODEX --validate-target ./.codex/

验证能力

--validate-target 是静态验证，不是运行时验证。它检查 TOML 是否可解析、MCP command 是否在 PATH 上、skill frontmatter 是否有 name 和 description、agent TOML 是否有必需字段、AGENTS.md 是否超过 32KB review threshold。

它不能证明 MCP server 一定能启动、hooks 行为与 Claude 等价、slash command 迁移后的 skill 可按原方式调用、子代理权限被硬性限制，或插件已经完成迁移。

核心策略总结

  1. 能结构化转换的就生成 Codex 原生文件。
  2. 语义不等价时保留提示，但标记人工复核。
  3. 高风险 provider runtime 行为不假装迁移成功。
  4. 先计划，后写入，通过 ConversionResult 与 DeploymentPlan 分离转换和部署。
  5. 报告驱动自愈：修复 MANUAL MIGRATION REQUIRED 后重新验证。

设计优点与局限

优点局限
边界清晰：可迁移、部分迁移、不能迁移分别处理。不是语义等价迁移器，不能证明运行时行为一致。
可审计：artifact 和 report item 都可追踪。YAML 解析是简化实现，不是完整 YAML parser。
安全默认：默认 merge，不删除旧 artifact。MCP JSON 使用 json.loads，注释或 trailing comma 可能失败。
保留上下文：不支持字段不会静默丢失。权限和 hooks 迁移存在明显语义差异。
模块化：每类迁移面独立，便于扩展。plugins 只报告，不自动迁移。

实践建议

优先检查 .codex/config.toml、.codex/hooks.json、.codex/agents/*.toml、.agents/skills/source-command-*/SKILL.md、所有 MANUAL MIGRATION REQUIRED 和 .codex/migrate-to-codex-report.txt。

只有在确认目标 Codex skills / agents 中没有手工维护内容，并且希望目标与源映射结果保持同步时，才使用 --replace。否则使用默认 merge。

结论

migrate-to-codex 的核心思想是：不要把跨 agent 平台迁移伪装成简单文件复制。它把能安全映射的部分落到 Codex 原生文件中；对不能安全映射的部分，保留上下文、写入人工复核块、生成报告，并要求验证。

scan -> plan -> doctor -> dry-run -> migrate -> review manual blocks -> validate -> runtime test

它的最佳定位是迁移加速器和迁移审计器，而不是完全自动兼容层。
