# Agent 使用、学习与大模型知识化技术总结

> 目标：把 AI 编程助手源码分析之外的经验，沉淀成后续使用 Agent、学习 Agent、大模型应用设计时可复用的知识框架。  
> 适用对象：日常使用 Claude Code / Codex / Cursor / Devin 类 Agent 的个人开发者、技术负责人、Agent 产品设计者。

## 1. 核心观点

Agent 的价值不只是“让模型替我写代码”，而是把大模型变成一个能在边界内观察、计划、执行、验证和复盘的工作系统。真正值得沉淀的知识不是某个提示词，而是以下几类能力：

- 如何把目标拆成 Agent 可执行的任务。
- 如何设计上下文，让模型知道该看什么、不该猜什么。
- 如何建立工具、权限、验证和反馈闭环。
- 如何把一次成功经验转成可复用流程、技能、模板和知识库。
- 如何识别大模型的能力边界，避免把不确定推理当成确定事实。

如果只学习“怎么问模型”，收益会很快到顶；如果学习“如何设计 Agent 工作系统”，收益会随着项目复杂度持续增长。

## 2. Agent 和普通 Chatbot 的根本区别

普通 Chatbot 主要处理语言任务：解释、总结、生成、翻译、问答。

Agent 则多了三个关键能力：

1. 环境感知：能读取代码、文件、终端输出、网页、数据库、MCP 资源等外部状态。
2. 行动能力：能编辑文件、运行命令、调用 API、创建任务、操作浏览器、触发工作流。
3. 闭环能力：能根据执行结果继续判断、修正计划、重试、验证和汇报。

因此，Agent 的学习重点应该从“prompt 写法”升级为“工作流设计”：

```text
目标定义 -> 上下文收集 -> 计划 -> 执行 -> 观察结果 -> 修正 -> 验证 -> 沉淀
```

## 3. 使用 Agent 前最重要的知识

### 3.1 目标要可观察

差的目标：

- 帮我优化一下这个项目。
- 看看这个代码有没有问题。
- 做一个更好的版本。

好的目标：

- 找出登录流程中导致 token 刷新失败的根因，并修复对应测试。
- 给 `src/server/auth` 增加 GitHub OAuth 回调失败场景的单元测试。
- 分析这个 AI 编程助手的核心架构，并输出 Markdown 与 HTML 文档。

Agent 最适合处理有明确验收条件的任务。目标越可观察，Agent 越不容易偏离。

### 3.2 上下文比提示词更重要

大模型不是靠“神奇提示词”变强，而是靠正确上下文降低不确定性。对代码 Agent 来说，关键上下文包括：

- 当前工作目录。
- 相关源码文件。
- 入口文件和调用链。
- 测试命令和失败输出。
- 约束：不能改哪些文件、不能引入哪些依赖、兼容哪些平台。
- 验收标准：什么输出、什么测试、什么行为算完成。
- 历史决策：为什么以前这么设计。

提示词负责表达任务，上下文负责决定答案质量。

### 3.3 Agent 需要边界

边界越清楚，Agent 越可靠：

- 文件边界：只改哪些目录。
- 行为边界：能不能提交、推送、删除、安装依赖。
- 风险边界：哪些命令需要确认。
- 产品边界：哪些功能不是本次目标。
- 验证边界：必须跑哪些测试，哪些测试可以跳过但要说明。

没有边界的 Agent 容易做“看起来有帮助但不受控”的事。

## 4. 高质量 Agent 任务的结构

一个优秀任务描述通常包含：

```text
目标：
  要完成什么。

背景：
  为什么要做，相关系统是什么。

范围：
  允许改哪里，不允许改哪里。

验收：
  什么行为、文件、测试或文档算完成。

约束：
  技术栈、风格、兼容性、安全要求。

输出：
  希望最终给出代码、报告、PR、文档还是命令结果。
```

示例：

```text
目标：修复用户导入 CSV 时空行导致解析失败的问题。
范围：只修改 src/importer 和对应测试，不改 UI。
验收：新增一个包含空行的 fixture，测试应证明空行被跳过，非空行仍正常解析。
约束：不引入新依赖，沿用现有 parser。
输出：修改代码并运行相关测试，最后说明改了什么。
```

## 5. 学习 Agent 的四层模型

### 5.1 第一层：会用

目标是提升个人效率。

需要掌握：

- 如何描述任务。
- 如何让 Agent 读代码再动手。
- 如何要求它给出计划。
- 如何要求它跑测试并报告证据。
- 如何打断错误方向。

核心能力：把模糊需求转成可执行任务。

### 5.2 第二层：会控

目标是让 Agent 稳定可靠。

需要掌握：

- 权限模式。
- 沙箱和危险命令。
- Git 工作区保护。
- 测试驱动或验证驱动。
- 失败时让 Agent 系统调试，而不是盲改。

核心能力：控制风险和结果质量。

### 5.3 第三层：会组织

目标是让 Agent 服务复杂工程。

需要掌握：

- 子代理拆分。
- 任务队列。
- 工作树隔离。
- 文档、计划、任务和记忆的分工。
- MCP、插件、技能、hooks 的组合。

核心能力：把 Agent 变成工程流程的一部分。

### 5.4 第四层：会设计

目标是构建自己的 Agent 系统。

需要掌握：

- 工具 schema 设计。
- 权限仲裁。
- 状态机和消息协议。
- 上下文压缩。
- 可观测性。
- 错误恢复。
- 人机协作界面。

核心能力：设计一个可扩展、可验证、可恢复的 Agent runtime。

## 6. Agent 工作流的知识沉淀方式

### 6.1 把一次成功变成模板

每次 Agent 成功完成复杂任务后，问三个问题：

- 这次任务的通用步骤是什么？
- 哪些上下文是必须提供的？
- 哪些验证命令证明结果可靠？

然后沉淀成模板。例如：

```markdown
## Bugfix 模板

1. 复现问题或定位失败测试。
2. 读取相关入口、调用方、测试。
3. 写一个失败测试。
4. 最小修改让测试通过。
5. 跑相关测试。
6. 总结根因、改动和验证证据。
```

### 6.2 把重复任务变成技能

适合变成技能的任务：

- 每次步骤相似。
- 需要稳定遵守规则。
- 依赖项目惯例。
- 容易因为遗漏步骤造成风险。

例如：

- 发布前检查技能。
- PR review 技能。
- 数据库迁移技能。
- 前端视觉验收技能。
- 安全审计技能。

技能不是“更长的提示词”，而是可复用操作规程。

### 6.3 把项目事实变成文档

适合进项目知识库的内容：

- 架构边界。
- 模块职责。
- 常用命令。
- 本地开发步骤。
- 数据流和调用链。
- 重要设计决策。
- 已知风险和排查方式。

不适合进长期知识库的内容：

- 临时 TODO。
- 一次性调试输出。
- 当前分支的短期状态。
- 可以从源码直接看出的琐碎事实。

### 6.4 把个人偏好变成记忆

适合记忆：

- 你偏好的沟通语言。
- 你喜欢先计划还是直接执行。
- 你常用的技术栈。
- 你不希望 Agent 做的行为。
- 你对验证、提交、文档的偏好。

不适合记忆：

- 项目当前代码结构。
- 某个 bug 的临时上下文。
- 密钥、token、隐私信息。
- 会快速过期的事实。

## 7. 大模型能力边界

### 7.1 模型擅长的事

- 结构化总结。
- 从代码中归纳模式。
- 根据错误输出提出假设。
- 生成计划和测试案例。
- 重构局部代码。
- 编写文档。
- 在多约束下权衡方案。

### 7.2 模型不可靠的事

- 记住最新事实。
- 准确猜测未读取的代码。
- 在没有执行结果时判断测试是否通过。
- 长时间保持所有细节不丢。
- 做高风险安全决策。
- 在缺少边界时自动选择最佳范围。

### 7.3 正确使用方式

把模型当成“强推理但需要证据的协作者”，而不是“全知执行器”。

实践规则：

- 让它先读文件，再改代码。
- 让它先复现，再修 bug。
- 让它先写验收，再实现。
- 让它跑命令，再声称完成。
- 让它说明不确定性，而不是编造确定结论。

## 8. Agent 与知识管理的关系

Agent 会放大知识管理质量。

如果项目没有清晰文档，Agent 会反复花 token 重新探索。

如果项目有高质量知识库，Agent 可以更快进入有效工作状态。

建议建立四类知识：

1. 项目知识：架构、模块、命令、部署、测试。
2. 流程知识：bugfix、review、release、incident、migration。
3. 决策知识：为什么选择某方案，放弃了什么。
4. 协作知识：团队风格、代码规范、验证要求。

Agent 最适合帮助维护这些知识，但需要人决定哪些知识值得长期保存。

## 9. Agent 使用中的常见反模式

### 9.1 只给目标，不给上下文

结果：模型猜测项目结构，容易改错地方。

改法：让它先探索入口、调用方、测试。

### 9.2 让 Agent 一次做太多事

结果：上下文膨胀，验证困难，错误难定位。

改法：按可独立验证的单元拆任务。

### 9.3 不要求验证

结果：代码看起来合理，但实际不能运行。

改法：任务描述中明确测试命令或验收方式。

### 9.4 把 Agent 当搜索引擎

结果：它可能用过期知识回答。

改法：最新事实、价格、文档、API 变更必须查源。

### 9.5 把权限开太大

结果：误删、误提交、误推送、误改配置。

改法：默认最小权限，危险操作显式确认。

### 9.6 让 Agent 自己定义完成标准

结果：它可能选择最容易完成的解释。

改法：用户或团队先定义验收标准。

## 10. 面向开发者的 Agent 使用清单

开始任务前：

- 目标是否具体？
- 是否明确范围？
- 是否给出相关目录或入口？
- 是否说明不可做事项？
- 是否有验收标准？

执行过程中：

- Agent 是否先读代码？
- 是否建立计划？
- 是否复现问题？
- 是否保持修改范围最小？
- 是否报告了阻塞和不确定性？

完成前：

- 是否运行了相关测试？
- 是否说明测试命令和结果？
- 是否列出改动文件？
- 是否说明未覆盖风险？
- 是否没有改动无关文件？

## 11. 面向团队的 Agent 落地建议

### 11.1 建立项目入口文档

每个项目至少应有：

- 如何启动。
- 如何测试。
- 核心目录。
- 代码风格。
- 常见任务流程。
- 禁止事项。

### 11.2 建立 Agent 工作规范

例如：

- 非 trivial 任务必须先计划。
- bugfix 必须先复现。
- 完成前必须验证。
- 不允许无确认执行 destructive 命令。
- 不允许擅自引入依赖。
- 不允许绕过失败测试。

### 11.3 建立可复用技能库

把团队高频任务做成技能：

- review。
- test writing。
- release notes。
- migration。
- incident analysis。
- dependency upgrade。

### 11.4 建立知识回填机制

每次任务结束后，判断是否需要更新：

- README。
- AGENTS.md / CLAUDE.md。
- 架构文档。
- 故障排查文档。
- 技能或模板。
- 决策记录。

Agent 不只是消耗知识，也应该生产和维护知识。

## 12. 技术上如何设计一个可靠 Agent

如果要自己设计 Agent 系统，核心组件包括：

### 12.1 消息协议

需要定义：

- user message。
- assistant message。
- tool_use。
- tool_result。
- system/attachment/progress。
- 错误和中断消息。

重点是保证协议可恢复、可重放、可压缩。

### 12.2 工具 ABI

每个工具至少要有：

- 名称和描述。
- 输入 schema。
- 输出 schema。
- 权限检查。
- 是否只读。
- 是否可并发。
- 是否 destructive。
- 结果序列化。
- UI 展示。

### 12.3 权限系统

需要支持：

- allow/deny/ask。
- 临时授权和持久授权。
- headless 自动拒绝。
- 高风险操作确认。
- 组织策略。
- hooks 或 policy 插件。

### 12.4 执行调度

需要处理：

- 并发安全。
- 中断。
- 超时。
- 取消。
- 失败恢复。
- 结果顺序。
- 长任务后台化。

### 12.5 上下文管理

需要处理：

- token 估算。
- prompt cache。
- 自动压缩。
- 压缩后恢复关键上下文。
- 文件结果落盘。
- 长会话 resume。

### 12.6 可观测性

需要记录：

- 每次模型请求。
- 工具调用。
- 权限决策。
- hook 输出。
- 错误类型。
- token/cost。
- 验证命令。

没有可观测性，Agent 系统无法稳定迭代。

## 13. 用目录法学习 Agent 架构

你给出的目录本身是一种很好的学习方法：先建立项目全景，再进入核心循环，最后看扩展机制和设计哲学。后续学习任何 Agent 项目，都可以复用这个目录法。

### 13.1 推荐目录模板

| 层级 | 要问的问题 | 典型章节 |
|---|---|---|
| 项目全景 | 这个项目解决什么问题，用什么技术栈，目录怎么组织？ | 项目概览、技术栈、目录结构 |
| 运行入口 | 程序如何启动，如何加载配置和能力？ | 启动流程、配置系统、feature flag |
| 核心循环 | 模型如何多轮观察、调用工具、处理结果？ | Agent Loop、工具调用、流式执行 |
| 状态管理 | 长对话、任务、记忆和恢复如何处理？ | Task、Compact、Memory、Session |
| 扩展机制 | 用户和团队如何扩展能力？ | Skill、Hooks、MCP、Plugins |
| 设计评价 | 架构亮点、风险和可复用经验是什么？ | 关键文件、设计哲学、工程启发 |

### 13.2 为什么目录法有效

- 它避免一开始陷进单个函数细节，先建立心智地图。
- 它能快速判断一个 Agent 项目的成熟度：是否有工具 ABI、权限系统、状态恢复、上下文压缩和扩展机制。
- 它适合沉淀知识库：每个目录项都能变成一篇短文或一张架构图。
- 它也适合评估竞品：不同 Agent 的差异通常不在“是否能调用模型”，而在 loop、tool、permission、state、extension 这些层。


## 14. 学习路线

### 阶段一：个人效率

- 学会让 Agent 解释代码。
- 学会让 Agent 写测试。
- 学会让 Agent 修小 bug。
- 学会要求验证证据。

### 阶段二：工程协作

- 学会写任务规范。
- 学会拆分多步骤任务。
- 学会用计划、任务和文档管理长上下文。
- 学会用 Git 工作树隔离风险。

### 阶段三：Agent 系统

- 学习 tool calling。
- 学习 MCP。
- 学习权限和沙箱。
- 学习 session replay。
- 学习上下文压缩。
- 学习插件和技能机制。

### 阶段四：组织落地

- 建立团队 Agent 使用准则。
- 建立项目知识库。
- 建立技能库。
- 建立验证和审计流程。
- 把 Agent 纳入 CI、review、release、incident 流程。

## 15. 最值得长期记住的原则

1. Agent 的可靠性来自上下文、边界和验证，不来自更长提示词。
2. 工具调用能力越强，权限和审计越重要。
3. 不要让模型猜代码；让它读代码。
4. 不要让模型声称完成；让它用证据证明完成。
5. 不要把所有信息都塞进记忆；区分项目文档、任务状态、长期偏好和临时上下文。
6. 好的 Agent 工作流应该能被复用、审查和改进。
7. Agent 是工作系统，不是单轮问答界面。

## 16. 总结

后续学习 Agent 和大模型，最有价值的知识不是某个产品按钮或某句 prompt，而是理解一个完整 Agent 系统如何把大模型能力工程化：

```text
模型推理能力
  + 正确上下文
  + 结构化工具
  + 权限边界
  + 执行反馈
  + 验证证据
  + 会话记忆
  + 知识沉淀
  = 可持续使用的智能工作系统
```

对个人来说，重点是把 Agent 当成高级工程协作者，并学会给它目标、上下文和验收。

对团队来说，重点是把 Agent 纳入流程、知识库和质量体系，而不是只把它当成代码生成器。

对产品或平台设计者来说，重点是构建安全、可恢复、可扩展、可观察的 Agent runtime。模型本身只是核心组件之一，真正的壁垒在它周围的工程系统。
