# 《Claude》阅读总结（tw93，2026-03-12）

原文链接：https://tw93.fun/2026-03-12/claude.html

## 一、核心判断

文章从实战角度指出：要把 Claude（或类似编码 Agent）用好，关键不在“提示词越长越好”，而在于把职责分层，并把关键规则从文字约定升级为工程约束。

## 二、关键观点

1. 配置文件应短而硬
- `AGENTS.md/CLAUDE.md` 不应变成知识仓库。
- 只放稳定硬约束：流程、边界、禁止项、验收口径。

2. 六层架构思维
- Context：常驻规则与任务上下文。
- Tools/MCP：可调用能力边界。
- Skills：可复用流程模块。
- Hooks：自动触发动作。
- Subagents：并行执行单元。
- Verifiers：完成真值判定。

3. 上下文污染是主要风险
- 工具返回、日志、冗长文档会挤压有效上下文。
- 需控制输出长度、只保留决策关键信息。

4. Skills 要模块化与渐进加载
- 技能主文件定义目标、边界、步骤。
- 细节拆到 references/scripts，避免主上下文过重。

5. 长任务必须有压缩与交接策略
- 任务越长，越需要明确 compact/handoff 结构。
- 压缩时必须保留：决策、改动、验证状态、待办风险。

## 三、对日常开发的直接建议

1. 先 Plan，再 Execute
- 复杂任务先确认方案与边界，再进入改动，减少返工。

2. 规则代码化
- 关键约束接入脚本/CI，不依赖“记得遵守”。

3. 让“完成”有客观证据
- 每次交付附可复现验证结果，而非口头完成。

4. 控制工具和插件复杂度
- 只启用当前任务必要工具，降低噪声与误触发。

## 四、可直接落地的项目结构

1. `AGENTS.md`
- 放流程和硬约束，不放冗长背景。

2. `MEMORY.md`
- 放长期偏好与决策摘要，跨会话可复用。

3. `HANDOFF.md`
- 固定模板记录当前状态，便于切会话接力。

4. `.agents/skills/`
- 技能按需加载，保持主上下文精简。

5. `scripts/verify.sh` + CI
- 形成“完成前必须通过”的自动闸门。

## 五、一句话总结

这篇文章的重点是：把 Claude 从“提示词玩法”升级为“工程化 Agent 系统”，通过分层职责、上下文治理、技能化流程和可执行验证，实现稳定可复用的开发能力。
